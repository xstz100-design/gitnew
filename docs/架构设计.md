# 开放式看货 + 智能清单导出 + Telegram 人工结算 —— 架构设计

> 本文档完整描述"线上选品自动化 → 人工结算闭环"的技术架构与业务下单流程。
> 适用场景：东南亚批发/B2B 轻量电商、社群团购、私域流量变现。

### 核心原则

| 维度 | 策略 |
| ---- | ---- |
| **鉴权** | 商品浏览/加购零鉴权；管理后台 JWT 鉴权 |
| **购物车** | 全量存储在 Pinia (persistedstate) → localStorage，服务器零状态 |
| **订单** | 不存在订单表。清单由前端生成，客服在 Telegram 人工处理 |
| **库存** | Go 后端是唯一权威源，前端实时只读查询 |
| **支付** | 无线上支付模块。由客服在 Telegram 确认到账 |
| **并发瓶颈** | 95% 请求为只读，Redis 缓存轻松承载万级 QPS |

### 3.3 手动出库接口——核心逻辑（伪代码）

```go
// POST /api/admin/stock-out
func StockOut(c *gin.Context) {
    var req StockOutRequest
    if err := c.ShouldBindJSON(&req); err != nil {
        c.JSON(400, Error("参数错误"))
        return
    }

    err := db.Transaction(func(tx *gorm.DB) error {
        for _, item := range req.Items {
            // 1. 锁定行级锁（悲观锁）或使用乐观锁
            var sku SKU
            if err := tx.Set("gorm:query_option", "FOR UPDATE").
                First(&sku, item.SKUID).Error; err != nil {
                return err
            }

            // 2. 校验库存
            if sku.Stock < item.Quantity {
                return fmt.Errorf("SKU %s 库存不足: 剩余 %d", sku.SKUCode, sku.Stock)
            }

            // 3. 扣减库存
            if err := tx.Model(&sku).
                Update("stock", gorm.Expr("stock - ?", item.Quantity)).
                Error; err != nil {
                return err
            }

            // 4. 写入库存变动日志
            tx.Create(&StockLog{
                SKUID:     item.SKUID,
                Type:      "out", // 出库
                Quantity:  item.Quantity,
                Note:      req.Note,
                Operator:  c.GetString("username"),
                CreatedAt: time.Now(),
            })
        }
        return nil
    })

    if err != nil {
        c.JSON(409, Error(err.Error()))
        return
    }

    // 5. 清除 Redis 缓存
    redis.Del(ctx, productCacheKeys(req.ProductIDs)...)

    c.JSON(200, OK("出库成功"))
}
```

**关键特性：**

- 使用 `pinia-plugin-persistedstate` 自动持久化到 `localStorage`
- 添加商品时携带当时库存快照，但生成清单前会二次校验
- 支持多种计价模式（零售件、打包、整箱）

### 4.3 清单生成核心代码示意

```javascript
// 生成采购清单（完整链路）
async function generatePurchaseList() {
  const cartStore = useCartStore()

  // 1. 库存二次校验
  const stockResult = await checkStocks(cartStore.items)
  const warnings = stockResult.filter(s => s.insufficient)

  if (warnings.length > 0) {
    // 提示用户库存不足项
    await showStockWarning(warnings)
    // 用户可以选择移除不足项或继续
    if (!userConfirm) return
  }

  // 2. 格式化文本清单
  const text = formatCartToText(cartStore.items)
  await navigator.clipboard.writeText(text)

  // 3. 生成图片（如果启用）
  if (featureFlags.exportImage) {
    const imageBlob = await cartToImage(cartStore.items)
    downloadImage(imageBlob)
  }

  // 4. 生成分享链接（可选）
  const shareLink = generateShareLink(cartStore.items)

  // 5. 跳转 Telegram
  window.location.href = TELEGRAM_BOT_LINK + '?start=' + shareLink
}
```

## 六、完整下单流程（业务视角）

### 第一阶段：线上选品（自动化，零门槛）

```text
Step 1 ── 免密准入
         用户通过分享链接或扫码进入商城
         → 无需注册/登录，直接看到商品
         → 路由无鉴权守卫，API 无鉴权拦截

Step 2 ── 浏览与加购
         用户浏览分类、商品详情
         → 选择规格（颜色/尺寸），点击"加入购物车"
         → 前端校验：本地数量 + 实时库存（只读查询）
         → 数据写入 localStorage，关闭页面不丢失

Step 3 ── 生成采购清单
         点击购物车底部【生成采购清单并联系客服】
         → 前端二次请求实时库存，标红库存不足项
         → 格式化文本清单（复制到剪贴板）
         → 可选生成长图（含缩略图、二维码）
         → 自动跳转到 Telegram 客服对话框
```

### 第二阶段：人工结算（灵活闭环）

```text
Step 4 ── 飞机对接
         用户在 Telegram 内发送清单/长图给客服

Step 5 ── 客服灵活报价（核心优势）
         ┌─ 老客户/批发 → 给批发折扣价
         └─ 新散客     → 给零售价
         → 客服人工计算运费（本地区域配送）
         → 回复总价："总共 XX 美金"

Step 6 ── 付款确认
         用户转账（红利/汇旺/现金）
         → 发送转账截图给客服
         → 客服确认到账

Step 7 ── 手动出库核销
         客服/仓管登录管理后台
         → 进入"手动出库"模块
         → 勾选对应商品/规格/数量
         → 点击【确认出库】
         → Go 后端事务：扣库存 → 写日志 → 清缓存
         → 前端商城实时库存同步更新 ✓

Step 8 ── 发货
         仓库拣货、打包、本地配送
         → 客服在 TG 上发送物流单号给用户（可选）
```

## 七、技术优势分析

### 7.1 并发能力

| 指标 | 传统电商 | 本架构 |
| ---- | -------- | ------ |
| 读写比 | 读 60% / 写 40% | 读 95% / 写 5% |
| 购物车 | 数据库写压力 | 零后端开销 |
| 订单 | 复杂状态机 + 锁 | 不存在 |
| 支付 | 回调 + 对账 | 不存在 |
| 单机 QPS | 500-1000 | 5000-10000+ |
| Redis 命中率 | 60-70% | 95%+ |

### 7.2 开发效率

- **后端接口数**: 仅需 ~15 个路由（传统电商需要 50+）
- **无订单状态机**: 省去 pending → paid → confirmed → shipping → completed → refund 等 6+ 状态流转
- **无支付集成**: 省去 Stripe/PayPal/支付宝 SDK 对接、webhook 签名验证、对账脚本
- **无并发死锁**: 唯一写操作是管理员手动出库，无并发竞争
- **前后端联调成本**: 降低 70% 以上

### 7.3 运维复杂度

- **无需消息队列**: 不处理异步订单超时、支付回调
- **无需定时任务**: 不释放超时未支付库存
- **无需数据一致性校验**: 无订单与支付的对账需求
- **部署简单**: Go 单二进制 + 静态文件 + Redis + DB

## 八、适用场景与注意事项

### 8.1 最佳适用场景

- **B2B 批发**: 客户批量采购，价格需要人工协商
- **私域社群团购**: Telegram/微信群里分享商品，客服统一接单
- **东南亚新兴市场**: 红利/汇旺等本地支付方式难以自动化对接
- **小规模精品商超**: 品类有限，不需要复杂的订单自动化

### 8.2 潜在瓶颈与规避

| 瓶颈 | 解决方案 |
| ---- | -------- |
| 客服人工处理上限 | 初期单人可处理；后期可分配给多个客服 TG 账号 |
| 库存并发写 | 手动出库频率极低（每天几十单），行级锁足够 |
| 前端购物车溢出 | localStorage 有 5-10MB 限制，正常情况完全够用 |
| 图片 CDN | 商品图片用 CDN 加速，后端 Go 不做图片处理 |

### 8.3 未来可平滑升级（无需重构）

```text
当前（零订单）                 未来（如需订单系统）
────────────                  ────────────────────
前端 localStorage 购物车   →   迁移到后端购物车表
TG 人工结算                →   接入 Stripe/汇旺 API
手动出库                   →   用户点击"下单"自动生成订单
无用户系统                 →   添加 JWT 用户注册/登录
```
